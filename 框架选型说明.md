# 框架选型说明

## 结论

本组选择 **Streamlit** 作为本周 Demo 的开发框架。

## 候选对比

| 框架 | 优点 | 缺点 | 是否适合本组 |
|------|------|------|--------------|
| **Streamlit** | 纯 Python 写页面，零前端基础也能上手；最快成型；社区资料多 | 复杂交互/自定义布局能力有限 | ✅ 选定 |
| Gradio | 适合模型输入输出演示，组件简单 | 复杂页面组织能力弱，多工具集成不如 Streamlit 顺手 | 备选 |
| Flask / FastAPI + HTML | 更工程化，能学后端路由和接口 | 需要自己写 HTML/CSS/JS，零基础开发量大 | 否 |
| React / Vue / Next + API | 能力上限高，前后端分离 | 对零基础门槛太高，一周内难跑通 | 否 |

## 选择理由（对应讨论标准）

1. **组员基础**：组内零前端基础，Streamlit 只用 Python 即可产出网页，学习成本最低。
2. **资料是否充足**：Streamlit 官方文档和中文教程丰富，遇到问题容易查。
3. **1 天内能否跑通**：`pip install` 后几行代码就能出页面，当天可见成果。
4. **是否适合本组功能**：我们的工具都是"输入文本 → 调用模型 → 显示结果"的简单交互，
   以及一个本地 CSV 预览。Streamlit 的 `text_area` / `file_uploader` / 多页机制完全够用。

## 取舍说明（验收时要能讲）

- 我们**放弃了前后端分离**：一周时间内，分离架构的收益（可扩展性）不如"快速做出能演示的东西"重要。
- 我们**用 Streamlit 多页（pages/ 目录）**实现"一个入口 + 多个工具菜单"，而不是手写路由。
- 对于 AI 调用，我们做了**真实 + 模拟双模式**，把"API 不通"这个最大风险提前消化掉，
  保证验收当天无论网络如何都能演示。
